Why it exists

Agents need more than chat and tickets.

Chat fades

Intent disappears

Long projects drift when the only memory is a transcript or a thread that future agents will not read completely.

Issues are local

Tickets go deep

GitHub issues are excellent for a step, but they are too narrow to carry the whole epic, its dependencies, and its direction.

Roles blur

Agents overreach

A worker starts planning, a tester starts rewriting, a parent agent mutates child docs. The system needs boundaries.

The root concept

Epic 0 holds the whole project together.

Epic 0 spine The root context: why the project exists, child-spine map, project state, cross-epic decisions.
›
Child spines Bounded delivery surfaces for Epic 2.3, 2.4, 2.5, or any other major track.
›
GitHub issues Executable tickets where one step can go deep into blockers, code, logs, and validation.

Epic 0 is why the project exists and how the moving parts fit. It is the root spine, not a giant ticket.

Delivery loop diagram

Epic workers manage execution without dirtying the spine.

Bound child spine Clean intent, state, acceptance, issue ledger, blockers, and handoff journal.
›
GitHub issues Executable ticket backlog with deep technical detail and subagent comments.
›
Ticket workers Implement one issue, write detail into the issue, and return PR/evidence.

The epic worker loops across this system until ready for human test, tester handoff, or a precise blocker.

Spine versus issues

The spine stays clean. Issues go deep.

EpicSpine

Intent, desire, context, fit, state

Mission, current state, child-spine map, decisions, dependencies, issue ledger, evidence links, and next action.

GitHub issue

Deep detail for one step

Code paths, implementation notes, blocker investigation, logs, stack traces, review discussion, and ticket-level validation.

If it only helps the current ticket worker, keep it in the issue. If it changes how future agents understand the epic, summarize it in the spine.

Goal seeking

Do not do one pass. Work toward a terminal state.

Epic 0

Keep the project coherent

Read child spines, roll up state, create child spines, bind workers, and keep pushing until the next human decision is explicit.

Epic worker

Deliver the scoped epic

Create issues, dispatch ticket workers, reconcile outputs, and loop until the epic is ready for human test.

Tester

Prove pass or fail

Run checks, record evidence, use bounded fix loops for small failures, and escalate scope changes to the planner.

Integration discipline

Separate branches. Fresh main. Frequent merges.

main The freshest integrated base for new agents, deployment, and human testing.
›
worker branch / worktree Each ticket worker gets isolated execution for one issue.
›
merge when ready Ready-for-test work returns to main; blockers stay visible in the issue and spine.

Do not let long-lived branches become hidden project state. If it is ready for test, merge it or record exactly why it cannot merge.

Write scope

Read broadly. Write narrowly.

Bound spine

The document you may edit

Each agent is bound to one spine. That spine is the writable operating surface for its role.

Referenced spines

Context by default

Parent, child, and sibling spines are read-only unless the agent is explicitly granted write authority.

Cross-spine drift

Propose, do not mutate

Record proposed changes in the bound spine, then route them to the owner of the target spine.
